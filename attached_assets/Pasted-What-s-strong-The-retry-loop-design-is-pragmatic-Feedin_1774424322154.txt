What's strong
The retry loop design is pragmatic. Feeding the error + sample rows back to Sonnet for a focused fix is exactly right — you're not asking the model to re-reason from scratch, you're giving it a diff to work with. The 2-retry cap with cost estimates shows you've thought about the operational envelope.
The proven_queries schema is well-designed. The success_count/fail_count with auto-deactivation at 3 failures is a clean self-healing mechanism. The upsert behavior (evolving toward the most recently successful SQL) is a subtle but important detail — it means the system naturally migrates away from deprecated tables without manual intervention.
The cost model table is a nice touch. It makes the ROI of Layer 2 immediately obvious to anyone reading this.
Where I'd push
The metric_type extraction feels like the weakest link and the doc somewhat acknowledges this. Keyword matching will break on synonyms ("earnings" works, "run rate" doesn't, "how much did they make" probably doesn't either). Given that this is the lookup key for your entire caching layer, a misclassification means you either miss a cache hit or worse, serve a cached query for the wrong metric. I'd consider spending the ~$0.01 on a Haiku call to canonicalize the metric type — it's a bounded classification task, latency would be minimal, and it protects the integrity of everything downstream.
The 3-failure deactivation window is a real UX problem that the doc underplays. Three users in a row get broken charts before the system self-corrects. You could add a staleness TTL — if a proven query hasn't been used in, say, 30 days, proactively re-validate it with a test execution (nightly job, cheap) rather than waiting for a user to hit the failure.
On the sanity checks — the doc correctly calls them "coarse," but I think the gap is larger than acknowledged. The three checks (all-zero, raw wei, majority-negative) catch maybe 40% of failure modes. The most insidious bugs are the ones that produce plausible-looking wrong numbers. Two additions worth considering: a magnitude-change check (if a proven query previously returned values in the $50M range and now returns $5B, flag it), and a cross-source validation (spot-check Dune results against DeFiLlama for protocols where both sources cover the metric).
Layer 3 reads like a brainstorm rather than architecture. That's fine if it's intentionally marked as "future thinking," but the doc's title says "Architecture Note" — I'd either cut it down to a two-line mention or flesh it out enough to be actionable. The open questions are good, but the "Possible Architecture" diagram doesn't add much beyond "run a nightly Claude call." The real design challenge is the evaluation loop (how do you know the prompt change helped?), and that's left unanswered.
Smaller things
The fallback cascade is described as "keyword-based routing," but the doc doesn't specify what happens when a metric doesn't match any keyword. Is there a true last-resort path, or does the user just get an error? Worth being explicit.
The doc mentions using Opus for initial plan generation and Sonnet for retries but doesn't explain the reasoning. If it's cost/latency, say so — otherwise a reader might wonder why you'd downgrade model quality at the exact moment the system is struggling.
No mention of observability. For a system that's designed to self-improve, you need to be logging the full lifecycle (request → cache hit/miss → LLM call → execution → sanity result → retry count → final outcome) in a way that's queryable. This is also what makes Layer 3 feasible eventually.
Bottom line
Layers 1 and 2 are solid and production-ready as described. The main risks are metric classification accuracy and the stale-query window. Layer 3 needs more work before it belongs in the same doc. The system would also benefit from a lightweight observability layer that feeds back into everything else.
