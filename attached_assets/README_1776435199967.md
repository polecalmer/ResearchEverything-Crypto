# Data-Source Brain

A vector-DB-backed knowledge store whose "facts" are the limitations, quirks, and semantic conventions of crypto data sources. Pipelines consult this brain before and during execution to avoid common errors — wrong field semantics, silent methodology changes, rate-limit surprises, coverage gaps, cross-source disagreements.

**Status**: Working prototype. CLI loop operational end-to-end on Mac Mini M4 / MacBook.

---

## Quick Start

```bash
cd data-source-brain
npm install
npx tsx brain/cli.ts seed all          # populate 62 facts across 5 sources
npx tsx brain/cli.ts consult "rate limits on DeFiLlama"
npx tsx brain/cli.ts inspect           # see brain stats
```

## Architecture

```
                ┌─────────────────────────────┐
                │        CLI / API Consumer    │
                └──────────┬──────────────────┘
                           │
              ┌────────────▼────────────────┐
              │     brain/api.ts            │
              │  consult() · observe()      │
              │  getFact() · getStats()     │
              └────────────┬────────────────┘
                           │
              ┌────────────▼────────────────┐
              │     brain/db.ts             │
              │  LanceDB vector store       │
              │  .brain-data/ on disk       │
              └────────────┬────────────────┘
                           │
              ┌────────────▼────────────────┐
              │   brain/embeddings.ts       │
              │   all-MiniLM-L6-v2 (local)  │
              │   384-dim embeddings        │
              └─────────────────────────────┘
```

**Data flow**: Seeders generate `Fact[]` from docs → `insertFacts()` embeds + stores in LanceDB → `consult()` embeds query + nearest-neighbor search → `observe()` deduplicates + stores runtime observations.

**Key files**:

| File | Purpose |
|------|---------|
| `brain/schema.ts` | Fact type definition, enums, dedupe key generation |
| `brain/embeddings.ts` | Local embedding via `@xenova/transformers` |
| `brain/db.ts` | LanceDB wrapper — insert, search, observe, stats |
| `brain/api.ts` | Clean public API re-exports |
| `brain/cli.ts` | Commander.js CLI — seed, consult, observe, inspect |
| `brain/seed/*.ts` | Per-source seeders (one file per data source) |

## Fact Schema

Every fact in the brain has these fields:

```typescript
interface Fact {
  fact_id: string;         // UUID
  source: Source;          // "defillama" | "coingecko" | "dune" | "allium" | "stonksonchain"
  scope: Scope;            // "source" | "endpoint" | "field" | "cross-source"
  scope_ref: string;       // e.g. "defillama:/protocols", "allium<->dune"
  category: Category;      // "rate_limit" | "auth" | "coverage" | "definition" | "freshness"
                           // | "reliability" | "schema" | "disagreement" | "other"
  content: string;         // Natural-language statement of the fact
  confidence: Confidence;  // "verified_doc" | "verified_runtime" | "observed_once"
                           // | "inferred" | "unverified"
  source_of_fact: string;  // URL or identifier — must always be populated
  observed_count: number;  // Incremented on re-observation
  created_at: string;      // ISO 8601
  last_seen_at: string;    // ISO 8601
  stale_at?: string;       // Optional expiry timestamp
}
```

The vector embedding is computed over `[source] [scope_ref] content` for retrieval context.

## CLI Reference

```bash
# Seed
npx tsx brain/cli.ts seed <source>         # seed one source
npx tsx brain/cli.ts seed all              # seed all five

# Consult
npx tsx brain/cli.ts consult "<query>"     # semantic search
npx tsx brain/cli.ts consult "<query>" --source defillama   # filter by source
npx tsx brain/cli.ts consult "<query>" --top-k 10           # control result count

# Observe
npx tsx brain/cli.ts observe <source> <scope_ref> <category> "<content>"
npx tsx brain/cli.ts observe defillama "defillama:/protocols" rate_limit \
  "DeFiLlama /protocols returned 429 at 15 rps" --source-of-fact "pipeline-run-001"

# Inspect
npx tsx brain/cli.ts inspect               # summary stats
npx tsx brain/cli.ts inspect <fact_id>     # single fact detail
```

## Three Core Operations

### 1. `seed(source)`
Reads documentation, generates facts, embeds them, inserts into LanceDB. Idempotent — re-running skips duplicates (deduped by hash of `source + scope_ref + content`).

### 2. `consult(query, source?, topK?)`
Embeds the query, runs nearest-neighbor search against the brain, returns ranked facts. Optional source filter and result count.

### 3. `observe(source, scopeRef, category, content, sourceOfFact)`
Records a runtime observation. Two-level dedup:
1. **Exact**: hash match on `(source, scope_ref, content)` → increment `observed_count`
2. **Semantic**: cosine similarity < 0.3 with same source + scope_ref + category → increment existing fact
3. Otherwise → insert as new fact with `confidence: "observed_once"`

## Development Guide

### Adding a new data source

1. Create `brain/seed/newsource.ts`:
```typescript
import type { Fact } from "../schema.js";
import { makeFact } from "./helpers.js";

export function seedNewSource(): Fact[] {
  return [
    makeFact({
      source: "newsource",       // must match the Source enum
      scope: "source",
      scope_ref: "newsource",
      category: "auth",
      content: "Description of the authentication model...",
      confidence: "verified_doc",
      source_of_fact: "https://docs.newsource.com/auth",
    }),
    // ... at least 10 facts
  ];
}
```

2. Add the source to `brain/schema.ts` SOURCES array:
```typescript
export const SOURCES = [
  "defillama", "coingecko", "dune", "allium", "stonksonchain",
  "newsource",  // <-- add here
] as const;
```

3. Register in `brain/seed/index.ts`:
```typescript
import { seedNewSource } from "./newsource.js";

const SEEDERS: Record<Source, () => Fact[]> = {
  // ... existing
  newsource: seedNewSource,
};
```

4. Run `npx tsx brain/cli.ts seed newsource` to test.

### Fact-writing guidelines

- **No guessing.** If the API docs don't specify something, set confidence to `"unverified"` and say so in the content.
- **Every fact must have a `source_of_fact`** — a URL, file path, or other traceable identifier.
- **Prefer narrow verified facts** over broad inferred ones.
- **One fact = one atomic claim.** Don't combine rate limits and auth in a single fact.
- **Include the "so what"** — not just what the API does, but what the gotcha is for a pipeline consuming it.

### Upgrading the embedding model

The embedding model is isolated in `brain/embeddings.ts`. To swap:

1. Change the model name in `getPipeline()` (e.g., `"Xenova/bge-small-en-v1.5"` for 384-dim, or `"Xenova/all-mpnet-base-v2"` for 768-dim)
2. Update `EMBEDDING_DIM` to match
3. **Delete `.brain-data/`** — embeddings are not compatible across models
4. Re-seed: `npx tsx brain/cli.ts seed all`

For production, you could swap to OpenAI `text-embedding-3-small` by replacing the `embed()` function with an API call. The DB layer doesn't care where embeddings come from.

### Upgrading the vector DB

The DB layer is isolated in `brain/db.ts`. To migrate to pgvector:

1. Replace `@lancedb/lancedb` with `drizzle-orm` + `pg` + `pgvector`
2. Create a Drizzle schema matching the `FactRow` interface (the reference repo's `systemLearnings` table is a good starting point)
3. Swap the `getTable()` / `insertFact()` / `consult()` implementations
4. The `brain/api.ts` public interface stays the same — consumers don't need to change

### Integrating with a pipeline

The brain is designed to be imported as a module:

```typescript
import { consult, observe } from "./brain/api.js";

// Before running a DeFiLlama pipeline:
const warnings = await consult("fetch protocol TVL from DeFiLlama", {
  source: "defillama",
  topK: 5,
});
// → Returns facts about slug resolution, timestamp format, etc.

// After a pipeline run encounters an issue:
await observe(
  "defillama",
  "defillama:/protocols",
  "rate_limit",
  "Got 429 response at 20 requests/second",
  `pipeline-run-${runId}`,
);
```

## What's Next — Roadmap for Further Development

### High priority (next iteration)

- **Live seeding from API docs** — Instead of hardcoded facts, fetch the actual docs at seed time (hit the Swagger/OpenAPI endpoints, parse, extract facts with an LLM). This makes seeding self-updating.
- **Pipeline integration** — Wire `consult()` into an actual data pipeline (e.g., the ResearchEverything-Crypto agent's `data-agent.ts`) so it checks the brain before every API call.
- **Auto-observe from pipeline errors** — When a pipeline gets a 429, a null response, or unexpected schema, automatically call `observe()` with the details.
- **Confidence promotion** — When `observed_count` reaches a threshold (e.g., 5), auto-promote from `observed_once` to `verified_runtime`.

### Medium priority

- **Staleness detection** — Periodically re-check facts against live API docs. Flag facts whose `source_of_fact` URL has changed. Use `stale_at` field.
- **Cross-source disagreement detection** — When two sources report different values for the same metric (e.g., DeFiLlama vs CoinGecko volume), create a `disagreement` fact automatically.
- **Fact decay** — Reduce confidence of facts that haven't been re-observed in N days.
- **HTTP API** — Wrap the CLI in an Express/Hono server so other services can consult the brain over HTTP.
- **Batch embedding** — Currently embeds one fact at a time. Batch embedding would speed up seeding ~10x.

### Low priority (production path)

- **Migrate to pgvector** — If deploying alongside the ResearchEverything-Crypto Supabase stack, use their existing Postgres + pgvector. The schema maps cleanly to a Drizzle table.
- **Upgrade embeddings** — Move to OpenAI `text-embedding-3-small` (1536-dim) for better retrieval quality. Requires API key but significantly improves semantic matching.
- **Multi-tenant** — Add user/team scoping so different pipelines can have private observations while sharing seed facts.
- **Web UI** — Simple dashboard showing fact counts, recent observations, staleness warnings.
- **Webhook ingestion** — Accept observations via webhook from running pipelines (currently CLI-only).
